Background

What is contract testing?

In a microservices or multi-consumer architecture, different applications depend on each other's APIs. When one side changes, the other can break without warning. Traditional integration tests catch this by deploying both sides and testing them together — but that's slow, flaky, and expensive.

Contract testing takes a different approach: each side tests independently against a shared contract that describes the expected API behavior. If both sides satisfy the contract, they'll work together in production — without ever needing to deploy together in a test environment.

The key idea

The consumer writes tests against a Pact mock server. Those tests generate a pact file — a JSON contract describing every HTTP interaction the consumer expects. The provider then verifies that it can satisfy every interaction in the contract by replaying the requests against the real API.

If both sides pass, they're compatible — without ever needing to be deployed together.

What a contract looks like

This is a real pact file generated during the spike — one interaction from the environment data source:

{
  "consumer": { "name": "KosliGoSDK" },
  "provider": { "name": "KosliAPI" },
  "interactions": [
    {
      "description": "a request to get environment production-k8s",
      "providerState": "environment production-k8s exists",
      "request": {
        "method": "GET",
        "path": "/api/v2/environments/test-org/production-k8s"
      },
      "response": {
        "status": 200,
        "body": {
          "name": "production-k8s",
          "type": "K8S",
          "description": "Production Kubernetes cluster",
          "include_scaling": true,
          "last_modified_at": 1234567890.123
        },
        // Matching rules: type-based, not exact values
        "matchingRules": {
          "$.body.name":        { "match": "type" },
          "$.body.type":        { "match": "type" },
          "$.body.description": { "match": "type" }
          // ...
        }
      }
    }
  ]
}

The matchingRules section is what makes this a contract rather than a fixture — "match": "type" means the provider must return a string, but the exact value doesn't matter. The API can change "production-k8s" to any string without breaking the contract.

Why we spiked it

Kosli has a growing multi-consumer landscape: the Terraform provider, CLI, Backstage plugin, MCP server, and a future operator all consume the same API. We spiked Pact to answer two questions:

  • Question A: Did Pact feel like a fit for the Terraform provider specifically?
  • Question B: Does the infrastructure investment plausibly amortize across the other consumers?

The Terraform provider was the spike vehicle because it's the structurally most favorable consumer: bounded surface, CRUD lifecycle, existing acceptance tests as reference. If Pact struggles here, it'll struggle elsewhere.

Section 4

Provider state experience

What is provider state?

A contract says: "when I GET /environments/org/production-k8s, the API returns a JSON object with these fields." But that only works if the environment actually exists. Provider state is Pact's mechanism for expressing that precondition.

On the consumer side, the test declares a state as a plain-English string — purely a label:

mockProvider.
    AddInteraction().
    Given("environment production-k8s exists").  // just a label
    UponReceiving("a request to get environment").
    WithRequest("GET", "/api/v2/environments/...").
    WillRespondWith(200, ...)

The mock server doesn't enforce this state — it always returns the canned response. The string just gets written into the pact file as metadata.

On the provider side, during verification, Pact looks up that string in a map of state handlers — functions that run before each interaction to make the precondition true:

StateHandlers: {
    "environment production-k8s exists": func(setup bool, s ProviderState) {
        // Called with setup=true BEFORE the interaction
        // Called with setup=false AFTER (teardown)
        if setup {
            apiClient.CreateEnvironment(ctx, &CreateEnvironmentRequest{
                Name: "production-k8s", Type: "K8S",
            })
        } else {
            apiClient.ArchiveEnvironment(ctx, "production-k8s")
        }
    },
}

With a stub server (what we used in the spike), these handlers are no-ops — the stub always returns the right shape regardless. With a real API, they become real code that creates and tears down test data.

Why this matters

Provider state conflates two concerns:

  • The contract itself — "given state X, the response has shape Y." This is the value add.
  • The state setup code — creating resources, managing dependencies, cleaning up. This is functional test infrastructure, not contract testing. It's also where the cost scales: each interaction needs its own handler, and dependent resources need multi-step setup.

Section 5

Failure message quality

Three failure scenarios tested. All produced clear, actionable messages:

Wrong type $.type → Expected 42 (Integer) to be the same type as 'K8S' (String)
Missing field $ → Actual map is missing the following keys: description
Wrong status expected 200 but was 404

JSONPath notation pinpoints the location. No stack traces or framework noise. If a contract breaks, the message tells us exactly what changed and where.

What is JSONPath?

JSONPath is a query language for addressing fields inside JSON — like XPath for XML. Pact generates it automatically from the matchers we define; we never write it ourselves.

"matchingRules": {
  "$.body.name":   { "match": "type" },  // must be a string
  "$.body.type":   { "match": "type" },  // must be a string
  "$.body.version": { "match": "type" }  // must be a number
}

$.body.name reads as: root ($) → response body (.body) → the name field.

Section 6

Cross-language relevance

Languages in scope

Three languages are relevant for Kosli SDK consumers of the public API (/api/v2/):

Go
Terraform provider, CLI. Spike tested with pact-go v2 (Rust FFI).
TypeScript
Backstage plugin, MCP server. Would use pact-js (native N-API binding).
Python
Future consumers, scripting, automation. Would use pact-python (Rust FFI).

What about the frontend?

The Kosli frontend is TypeScript/React but server-rendered via HTMX + Jinja2 templates. It communicates exclusively with internal Flask UI routes (returning HTML fragments), not the public /api/v2/ REST API. Contract testing between front- and backend would target a different API surface — the internal UI routes — and is a separate question from this spike.

Portability of pact files

Pact files are fully portable across all three languages. Matchers ("match": "type", "match": "regex") are Pact specification standard — pact-go, pact-js, and pact-python all generate and consume the same format. Field names are language-neutral snake_case from the API.

Provider state strings are plain English, shareable across SDKs. In production, state handlers live on the provider side — the API team writes them once.

Bottleneck: per-SDK native dependency The reusability bottleneck is not the pact files — it's that each Pact library requires a native binary installed on every machine. CI setup on Linux is straightforward (wget + copy to /usr/local/lib), though no official GitHub Action exists. Local development on macOS requires sudo to install and DYLD_LIBRARY_PATH at runtime.

Section 7

Open questions

Issues surfaced during the spike that would need answers before production adoption:

  1. CI / Infrastructure

    CI runner setup: Every Linux runner needs the FFI library installed (wget + copy to /usr/local/lib). No official GitHub Action exists. Local dev on macOS additionally requires DYLD_LIBRARY_PATH.
  2. Developer experience

    Onboarding friction: sudo pact-go install required for local development. This adds a native dependency to what is currently a zero-native-deps Go toolchain.
  3. Pact limitation

    Nullable types: How to express "null OR number" (e.g., last_reported_at)? Pact V2 doesn't support union types. V3 adds a null matcher and OR combinator.
  4. Pact limitation

    Empty arrays: matchers.EachLike(..., 0) not allowed — Pact forces min 1 element. Problem for fields like policies that can legitimately be empty.
  5. Pact limitation

    Multipart/form-data: Can't be matched on request body in Pact V2. V3+ adds multipart support. ~50% of CRUD interactions for custom attestation type have no request body contract as a result.
  6. Code quality

    Client bypass: CreateCustomAttestationType bypasses doRequest() and calls httpClient.Do() directly — skipping retry and auth logic. Should be refactored independently of Pact. Tracked in #198.
  7. CI / Infrastructure

    State handler scaling: Provider state handler count scales linearly with interactions. For full API coverage (~30+ endpoints), that's significant plumbing — especially for resources with dependencies that need multi-step setup.

Decision

Wait with Pact on the consumer side until the first SDK exists. Contracts are consumer-driven, so the most reasonable provider-side work is to get familiar with Pact until the first contract has been created. When an SDK is built, that’s the trigger to write consumer pact tests and stand up provider verification.

Pact works mechanically and its failure messages are excellent. The value is narrow (shape drift only) and the infrastructure cost is real — but the SDK model makes amortization viable, and shifting integration checks out of e2e into contract tests would give us faster, more focused feedback.

Shifting integration feedback left Today we rely on long-running e2e tests to catch conflicts between the API and clients. Contract testing would give us that signal earlier — in CI, on every SDK commit — with confidence that a given SDK version satisfies the API contract. This frees e2e tests to focus on what they’re actually good at: testing functional behavior end-to-end, rather than also serving as integration compatibility checks.

Key risks

High risk
Will the API team pick up Pact?

Contract testing is two-sided. Pact files are inert unless the provider runs verification in CI. Without provider-side buy-in, Pact reduces to documentation with extra steps. Requires the API team to run pact verify, write state handlers, and agree on a failure process.

Medium risk
Pact Broker infrastructure

Pact files must flow from consumer CI to provider CI. Options: PactFlow (SaaS, paid, zero infra), self-hosted broker (Docker + Postgres), or no broker (loses "can I deploy?" workflow). The broker is not optional for the full cross-team workflow.

Suggested next steps

1
Agree on contract testing investment
Align across teams on whether contract testing with Pact is the right approach going forward. This is a two-sided investment — both consumer and provider teams need to commit for the full value to materialize.
2
Create SDKs with consumer contract tests
SDK development will happen regardless of this spike, but we should have at least one stable SDK (e.g., consumed by terraform-provider-kosli) before investing in consumer-side contract testing. When the SDK boundary is established, add pact consumer tests to the SDK's CI.
3
Investigate contract publishing
Evaluate Pact Broker options: PactFlow (SaaS, zero infra), self-hosted (Docker + Postgres), or a simpler artifact-sharing approach. The broker enables the "can I deploy?" workflow and tracks verification results across teams.
4
Invest in provider-side verification
Stand up pact verify in the API's CI pipeline: state handlers, verification against real API, and a process for handling contract failures. This is where the contract testing loop closes.
