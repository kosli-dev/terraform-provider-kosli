Background

What is contract testing?

In a microservices or multi-consumer architecture, different applications depend on each other's APIs. When one side changes, the other can break without warning. Traditional integration tests catch this by deploying both sides and testing them together — but that's slow, flaky, and expensive.

Contract testing takes a different approach: each side tests independently against a shared contract that describes the expected API behavior. If both sides satisfy the contract, they'll work together in production — without ever needing to deploy together in a test environment.

The key idea

The consumer writes tests against a Pact mock server. Those tests generate a pact file — a JSON contract describing every HTTP interaction the consumer expects. The provider then verifies that it can satisfy every interaction in the contract by replaying the requests against the real API.

If both sides pass, they're compatible — without ever needing to be deployed together.

What a contract looks like

This is a real pact file generated during the spike — one interaction from the environment data source:

{
  "consumer": { "name": "KosliGoSDK" },
  "provider": { "name": "KosliAPI" },
  "interactions": [
    {
      "description": "a request to get environment production-k8s",
      "providerState": "environment production-k8s exists",
      "request": {
        "method": "GET",
        "path": "/api/v2/environments/test-org/production-k8s"
      },
      "response": {
        "status": 200,
        "body": {
          "name": "production-k8s",
          "type": "K8S",
          "description": "Production Kubernetes cluster",
          "include_scaling": true,
          "last_modified_at": 1234567890.123
        },
        // Matching rules: type-based, not exact values
        "matchingRules": {
          "$.body.name":        { "match": "type" },
          "$.body.type":        { "match": "type" },
          "$.body.description": { "match": "type" }
          // ...
        }
      }
    }
  ]
}

The matchingRules section is what makes this a contract rather than a fixture — "match": "type" means the provider must return a string, but the exact value doesn't matter. The API can change "production-k8s" to any string without breaking the contract.

Why we spiked it

Kosli has a growing multi-consumer landscape: the Terraform provider, CLI, Backstage plugin, MCP server, and a future operator all consume the same API. We spiked Pact to answer two questions:

  • Question A: Did Pact feel like a fit for the Terraform provider specifically?
  • Question B: Does the infrastructure investment plausibly amortize across the other consumers?

The Terraform provider was the spike vehicle because it's the structurally most favorable consumer: bounded surface, CRUD lifecycle, existing acceptance tests as reference. If Pact struggles here, it'll struggle elsewhere.

Section 2

Integration shape

Pact tests run as a parallel suite alongside existing tests. They import pkg/client, point it at Pact's mock server, and exercise client methods directly. Existing unit tests and acceptance tests are untouched.

Embedding Pact in the Terraform resource.Test framework is not viable — the framework controls the provider lifecycle and HTTP client internally, with no hook to inject a mock server.

Section 4

Provider state experience

Pact's provider state mechanism conflates two concerns:

  • The contract — "given state X, the response has shape Y." This is the value add.
  • The state setup — "create this environment so it exists when Pact replays the GET." This is functional test infrastructure, not contract testing.

The full Pact value — catching real API drift — only materializes when verifying against the real API. That's the most expensive mode. The acceptance tests get dependency orchestration for free via Terraform's dependency graph — but that advantage is Terraform-specific. SDK consumers don't have a dependency graph, so this cost gap disappears at the SDK level.

Section 5

Failure message quality

Three failure scenarios tested. All produced clear, actionable messages:

Wrong type $.type → Expected 42 (Integer) to be the same type as 'K8S' (String)
Missing field $ → Actual map is missing the following keys: description
Wrong status expected 200 but was 404

JSONPath notation pinpoints the location. No stack traces or framework noise. If a contract breaks, the message tells us exactly what changed and where.

Section 6

Cross-language relevance

Pact files are fully portable. Matchers ("match": "type", "match": "regex") are Pact specification standard — pact-js, pact-python, pact-jvm all generate and consume the same format. Field names are language-neutral snake_case from the API.

Provider state strings are plain English, shareable across SDKs. In production, state handlers live on the provider side — the API team writes them once.

The reusability bottleneck is not the pact files — it's the per-SDK native library dependency. pact-go wraps a 13MB Rust FFI binary. CI setup on Linux is straightforward (wget + copy to /usr/local/lib), though no official GitHub Action exists. Local development on macOS requires sudo to install and DYLD_LIBRARY_PATH at runtime.

Section 7

The "what about" list

  1. CI setup: every Linux runner needs the FFI library installed (wget + copy to /usr/local/lib). No official GitHub Action exists. Local dev on macOS additionally requires DYLD_LIBRARY_PATH.
  2. Developer onboarding: sudo pact-go install required. How does that sit with the current zero-native-deps Go toolchain?
  3. Nullable types: how to express "null OR number" (e.g., last_reported_at)? Pact V2 doesn't support union types. V3 adds a null matcher and OR combinator.
  4. matchers.EachLike(..., 0) not allowed — Pact forces min 1 element. Problem for fields like policies that can be empty arrays.
  5. Multipart/form-data can't be matched on request body in Pact V2. V3+ adds multipart support via the Rust FFI. ~50% of CRUD interactions for custom attestation type have no request body contract.
  6. CreateCustomAttestationType bypasses doRequest() — should be refactored. Tracked in #198.
  7. Provider state handler count scales linearly. For ~30+ endpoints, that's significant plumbing.

Decision

Wait with Pact on the consumer side until the first SDK exists. Contracts are consumer-driven, so the most reasonable provider-side work is to get familiar with Pact until the first contract has been created. When an SDK is built, that’s the trigger to write consumer pact tests and stand up provider verification.

A note on the spike context: This spike used the Terraform provider as the test vehicle, but the learning points toward Pact at the SDK level. Some findings are Terraform-specific — for example, the acceptance tests get dependency orchestration for free via Terraform’s dependency graph, making Pact’s state handlers feel like extra cost. That advantage disappears for SDK consumers, which don’t have a dependency graph. The scores below reflect what we observed in the spike; at the SDK level, authoring cost and infrastructure overhead would look more favorable.

Key risks

High risk
Will the API team pick up Pact?

Contract testing is two-sided. Pact files are inert unless the provider runs verification in CI. Without provider-side buy-in, Pact reduces to documentation with extra steps. Requires the API team to run pact verify, write state handlers, and agree on a failure process.

Medium risk
Pact Broker infrastructure

Pact files must flow from consumer CI to provider CI. Options: PactFlow (SaaS, paid, zero infra), self-hosted broker (Docker + Postgres), or no broker (loses "can I deploy?" workflow). The broker is not optional for the full cross-team workflow.

ROI position

Where does Kosli sit on the ROI spectrum?

Low ROI — stable API, few consumers High ROI — rapid API change, many SDKs

Position this marker based on how rapidly the Kosli API is evolving and how many SDK consumers are planned.

Bottom line

Pact works mechanically and its failure messages are excellent. The value is narrow (shape drift only) and the infrastructure cost is real — but the SDK model makes amortization viable. The trigger is the first stable SDK.

Suggested next steps

1
Agree on contract testing investment
Align across teams on whether contract testing with Pact is the right approach going forward. This is a two-sided investment — both consumer and provider teams need to commit for the full value to materialize.
2
Create SDKs with consumer contract tests
SDK development will happen regardless of this spike, but we should have at least one stable SDK (e.g., consumed by terraform-provider-kosli) before investing in consumer-side contract testing. When the SDK boundary is established, add pact consumer tests to the SDK's CI.
3
Investigate contract publishing
Evaluate Pact Broker options: PactFlow (SaaS, zero infra), self-hosted (Docker + Postgres), or a simpler artifact-sharing approach. The broker enables the "can I deploy?" workflow and tracks verification results across teams.
4
Invest in provider-side verification
Stand up pact verify in the API's CI pipeline: state handlers, verification against real API, and a process for handling contract failures. This is where the contract testing loop closes.
